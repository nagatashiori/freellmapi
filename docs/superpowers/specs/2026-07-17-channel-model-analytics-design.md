# 渠道 × 模型 Analytics 设计

## 目标

让运营者能清楚地看到一次调度实际调用的渠道和模型，能把错误归为可处理的原因，并以真实流量的“渠道 × 模型”质量基线判断后续人工调度策略。

## 范围

- 调度链总览显示请求模型、最终实际模型和最终渠道；每一步显示实际渠道、实际模型、尝试序号、状态、耗时与分类后的失败原因。
- 增加 `GET /api/analytics/channel-model-baselines`。它按 `(platform, model_id)` 聚合真实调用，排除 `request_type='probe'`；返回样本量、成功率、成功请求平均延迟、P50/P95、成功请求 TTFB、成功输出的 token/s、错误数与最后调用时间。
- 错误分类改为集中式、确定性的分类函数。分类包括：限流/配额、鉴权、权限、模型不存在、请求或能力不兼容、超时、连接/DNS/TLS、上游 5xx、上游不可用、流中断、网关路由和未知上游错误。
- `Other` 仅在原始错误为空时显示为“未记录错误”；其他无法识别的文字显示为“未知上游错误”，并保留原文供排查。

## 数据口径

- `platform` 是渠道（可理解为供应商/运营商入口），`model_id` 是实际调用的模型；两者合起来才是可比较的路由对象。
- 质量基线只使用真实流量（`request_type != 'probe'`），因为探测的输入、token 和超时模式不同；探测继续服务 v10 的实时健康排序。
- 延迟、P50、P95、TTFB 和 token/s 仅由成功调用计算；成功率与错误数使用同一组合的全部真实尝试。
- 本次仅呈现和解释基线，不修改 `profile_models` 的人工 priority，也不自动改写 runtime 排序。

## 兼容与验证

- 不改数据库 schema：`requests` 和 `routing_events` 已具备所需字段。
- 为 API 增加回归测试：排除 probe、正确分组、成功指标不混入失败值、错误分类完整。
- 前端仅消费新增端点和已有 trace 字段；通过 TypeScript build 与 Analytics 路由测试验证。
